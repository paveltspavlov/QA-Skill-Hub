Agents, Skills & Prompt Templates

This hub catalogues three kinds of ready-to-use AI building blocks for QA and development work. All of them are plain Markdown files that live in this repository — browse them here and copy what you need, or take the files straight from the Agents/, Skills/ and Prompts/ folders.

⚙

Agents

Autonomous, multi-step workers for an agentic tool (Claude Code, Copilot agent mode, Cursor). An agent reads your repo, runs commands, and proposes changes — e.g. the Commit Message Writer inspects your staged diff and writes a Conventional Commit.

★

Skills

Focused specialists you install once in a project and invoke on demand. Each skill packages a role, a tuned prompt, a deliverable shape, and a quality checklist for one job — e.g. generating ISTQB-aligned test cases from a user story.

⚡

Prompt Templates

Zero-install, copy-paste prompts for one-off tasks. Fill in the [PLACEHOLDERS] and paste into any chat AI — Claude, ChatGPT, Copilot Chat, Gemini. Nothing is saved in your project.

▤

Model Context Protocol (MCP)

MCP is an open standard for connecting LLMs to tools, data, and other systems. Think of it as USB-C for AI — one wire, many devices. The four sections below explain what MCP is, how it works, how it compares to other LLM–tool APIs, and how to build your own MCP server.

1. What is MCP?

The Model Context Protocol is an open, JSON-RPC–based protocol introduced by Anthropic in late 2024. It standardises how AI applications (hosts like Claude Desktop, Claude Code, Cursor, Zed) discover and call tools, read resources, and use prompt templates exposed by external servers — without each integration being one-off glue code.

★

Open Standard

Vendor-neutral spec with SDKs for Python, TypeScript, Go, Rust, Java, C#. Anyone can build a server; any compliant host can use it.

⚙

Three Primitives

Tools (LLM-invoked actions), Resources (read-only data the model can pull in), Prompts (reusable templates the user can pick).

⇄

JSON-RPC 2.0

Bidirectional messages over stdio, Server-Sent Events, or Streamable HTTP. Includes capability negotiation, lifecycle, and progress events.

⌂

Host ⇄ Client ⇄ Server

Host runs the LLM. Client (one per server) speaks MCP. Server exposes capabilities. Boundaries keep permissions and trust scoped per integration.

2. How MCP Works

The host (e.g. Claude Desktop) spawns an MCP client for each configured server. Client and server perform a JSON-RPC handshake, then the host can list and call capabilities on demand:

┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ HOST │ │ CLIENT │ │ SERVER │ │ (Claude app) │ ◄────► │ (per-server │ ◄────► │ (your code: │ │ + LLM │ │ MCP shim) │ │ filesystem, │ │ │ │ │ │ GitHub, DB) │ └──────────────┘ └──────────────┘ └──────────────┘ ▲ ▲ │ │ └─── user prompt ──────► tools/call ──────► business logic ◄── result ◄──

Lifecycle:

  1. initialize — client and server exchange protocol version + capabilities (tools, resources, prompts, sampling, roots).
  2. tools/list, resources/list, prompts/list — host fetches catalog. Tool descriptions are sent to the LLM as part of its system prompt.
  3. tools/call — when the LLM emits a tool call, the host forwards it to the server, which executes and returns content.
  4. notifications/* — server can push tools/list_changed, progress, log events; client can cancel.
  5. shutdown — clean teardown.

Transports:

  • stdio — server is a child process, messages over stdin/stdout. Best for local tools.
  • Streamable HTTP — single endpoint, supports resumable streams. Best for remote/multi-tenant servers.
  • SSE (legacy) — older HTTP+Server-Sent-Events transport, being phased out in favour of Streamable HTTP.

3. Other APIs for Connecting LLMs to Tools

MCP is one of several approaches. Each makes different trade-offs about scope (single LLM call vs. long-running agent), discovery (static vs. dynamic), and transport.

Anthropic native tool use (Messages API)

Tools are declared inline in each API call. The model returns a tool_use block; your code executes and replies with tool_result. No separate server — tools live in your application.

response = client.messages.create(
    model="claude-opus-4-7",
    tools=[{
        "name": "get_weather",
        "description": "Get current weather for a city.",
        "input_schema": {
            "type": "object",
            "properties": {"city": {"type": "string"}},
            "required": ["city"]
        }
    }],
    messages=[{"role": "user", "content": "Weather in Sofia?"}]
)

OpenAI function calling / tools

Conceptually identical to Anthropic native tools: JSON-Schema function declarations passed per request. Mature ecosystem, single vendor.

resp = openai.chat.completions.create(
  model="gpt-4o",
  tools=[{
    "type": "function",
    "function": {
      "name": "get_weather",
      "parameters": {"type": "object", "properties": {"city": {"type":"string"}}}
    }
  }],
  messages=[{"role":"user","content":"Weather in Sofia?"}]
)

Google Agent-to-Agent (A2A)

Open protocol announced by Google (2025) for agent-to-agent collaboration, not LLM-to-tool. Each agent publishes an AgentCard at /.well-known/agent.json describing its skills; other agents delegate Tasks to it via JSON-RPC over HTTPS, with streaming updates and artifacts.

// Agent Card (served at /.well-known/agent.json)
{
  "name": "TravelPlannerAgent",
  "description": "Books flights and hotels.",
  "url": "https://travel.example.com/a2a",
  "skills": [
    { "id": "book-flight",
      "description": "Book a flight given origin, destination, dates" }
  ],
  "capabilities": { "streaming": true, "pushNotifications": true }
}

// Caller delegates a task
POST /a2a  { "method": "tasks/send",
             "params": { "id": "t-123",
                         "message": { "role": "user",
                                      "parts": [{"text": "Book SOF→LHR May 12"}] } } }

MCP vs A2A: MCP exposes tools/data to a single LLM in a single host; A2A coordinates autonomous agents across organizations. They are complementary, not competing — an A2A agent often uses MCP internally to reach its own tools.

LangChain / LlamaIndex tools

Library-level abstraction. Tools are Python/TS objects bound to an agent loop run by your code. Great for prototyping; tightly coupled to the framework.

from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return weather_api.lookup(city)

agent = create_react_agent(llm, tools=[get_weather])

OpenAPI-as-tools (ReAct / Toolformer style)

Feed an OpenAPI/Swagger spec to the LLM; it picks endpoints to call. Zero protocol overhead but no permission model and no streaming — the LLM is essentially curling raw HTTP.

# Spec is summarised into the prompt
POST /pets   body: { name, species }   → 201 Pet
GET  /pets/{id}                         → 200 Pet

# Model output (parsed by your runtime)
{ "action": "GET", "path": "/pets/42" }

MCP vs REST

REST is a general-purpose request/response style for client–server data access. MCP is a protocol on top of JSON-RPC built specifically for an LLM to discover and invoke capabilities at runtime. They solve different problems and often live side-by-side: an MCP server frequently wraps a REST API and re-exposes it in a form the model can use safely.

AspectRESTMCP

Message styleHTTP verbs + URIs (resource-oriented)JSON-RPC 2.0 methods (action-oriented)

AuthOAuth, bearer, mTLS — between servicesPer-host trust boundary; remote MCP adds OAuth on top
StreamingOptional (SSE / WebSocket bolt-on)First-class: progress events, partial results, cancellation
ErrorsHTTP status + body shape variesStructured { isError: true, content } the model can reason about

Rule of thumb: if a human or another service is calling it, expose REST. If an LLM is calling it, expose MCP — and have the MCP server call your REST API underneath.

4. How to Create an MCP Server

The fastest path is the official Python SDK (pip install mcp) using the FastMCP helper. The example below exposes one tool that the LLM can call.

  1. Install the SDK — pip install "mcp[cli]" (the cli extra ships the MCP Inspector for testing).
  2. Write the server — one file is enough for a working server.
# server.py
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("weather-server")

@mcp.tool()
def get_weather(city: str) -> str:
    """Return the current weather for a city."""
    # Replace with a real API call.
    return f"It's 18°C and clear in {city}."

@mcp.resource("weather://recent")
def recent_lookups() -> str:
    """Most recent cities the user looked up."""
    return "Sofia, London, Berlin"

if __name__ == "__main__":
    mcp.run(transport="stdio")
  1. Register the server with a host. For Claude Desktop, edit claude_desktop_config.json:
{
  "mcpServers": {
    "weather": {
      "command": "python",
      "args": ["/abs/path/to/server.py"]
    }
  }
}
  1. Test with the Inspector before wiring it into a host:
npx @modelcontextprotocol/inspector python server.py
# → Opens a browser UI to list tools, call them, and inspect raw JSON-RPC.
  1. Restart the host. The new tool now appears in Claude's tool palette and the model can call it.

TypeScript equivalent

// server.ts — npm i @modelcontextprotocol/sdk zod
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "weather-server", version: "1.0.0" });

server.tool(
  "get_weather",
  { city: z.string().describe("City name") },
  async ({ city }) => ({
    content: [{ type: "text", text: `It's 18°C and clear in ${city}.` }]
  })
);

await server.connect(new StdioServerTransport());

Production checklist: declare a JSON Schema for every tool input, return structured errors with isError: true, scope side-effects to a clearly-named tool (don't hide writes inside a "get"), add timeouts, log to stderr (stdout is the protocol channel), and pin the protocol version your SDK was tested against.
